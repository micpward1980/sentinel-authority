Conformance Requirements

ODDC certification requires a system to satisfy five gates: a declared or auto-discovered operational domain, demonstrated operational stability, runtime-enforced boundaries via ENVELO Interlock, a tamper-evident evidence record, and a clear revocation pathway.

01
ODD Declaration

The operational design domain must be formally declared or auto-discovered through ENVELO's adaptive learning engine. Fifteen boundary types are supported. No domain-specific configuration is required. See R-01b.

02
Stability

The system must demonstrate 72+ cumulative hours of stable operation within its declared ODD under monitored conditions.

03
Interlock

All three ENVELO Interlock enforcement tiers must be present and verified: Tier 1 self-correction, Tier 2 Minimum Risk Condition, Tier 3 hard halt.

04
Evidence

A tamper-evident, SHA-256 hash-chained audit record must be generated for all enforcement events and human decisions. Database triggers prevent modification. See R-06, R-06a.

05
Revocation

The applicant must execute a Conformance Agreement that specifies conditions under which the attestation may be revoked or suspended.

Detailed Requirements
  • R-01ODD boundaries must be defined in machine-readable format compatible with ENVELO Interlock's enforcement engine, or auto-discovered and operator-approved prior to enforcement activation. Declared boundaries must include at minimum: (a) at least one quantitative operational parameter with defined limit values, (b) a defined geographic, environmental, or operational scope, and (c) explicit identification of conditions under which enforcement activates. Boundaries defined solely by time window, administrative classification, or other non-operational criteria do not satisfy this requirement.
  • R-01aODD declarations are subject to Pre-CAT-72 Audit Control Review. Sentinel Authority will return declarations that are insufficiently specified, internally inconsistent, or that define boundaries so narrow as to render the verification period non-representative of actual operational conditions. Revision history of ODD declarations is retained and published in the certificate registry.
  • R-01bThe ENVELO discovery engine supports the following boundary types, each of which may be auto-discovered from observed system behavior or manually declared:
    (a) Numeric range constraints (b) Categorical value sets (c) Geographic geofences (d) Temporal operating windows (e) Rate-of-change limits (f) Compound conditional rules (g) Multi-modal distributions (h) Sequence / prerequisite enforcement (i) Boolean sensor health (j) Connectivity / heartbeat gap limits (k) Cumulative threshold monitoring (l) Statistical rolling-window constraints (m) Frequency / event-count limits (n) Baseline drift detection (o) Multi-variable logical conditions
    All boundary types are evaluated synchronously at execution time.
  • R-02The system must complete a minimum of 72 cumulative hours of monitored operation without boundary violation during the CAT-72 assessment period.
  • R-03ENVELO Tier 1 (self-correction) must engage automatically on ODD approach with no human intervention required.
  • R-04ENVELO Tier 2 (Minimum Risk Condition) must activate on ODD boundary breach and bring the system to a defined safe state.
  • R-05ENVELO Tier 3 (hard halt at the ENVELO wall) must function as an absolute override that cannot be disabled by operator or software command.
  • R-06All enforcement events must be logged with timestamps, positional context where applicable, and cryptographic hash chains linking each event to the prior record.
  • R-06aThe following actions are independently logged in the tamper-evident audit chain:
    Application submission Application approval / rejection Reviewer assignment Test creation & start Specification confirmation Test completion with result Certificate issuance Certificate auto-suspension Certificate reinstatement API key generation / revocation Boundary report submission Surveillance config changes User authentication events Profile modifications
    Each entry records the acting user, timestamp, resource identifier, and action-specific details. Database-level triggers prevent UPDATE or DELETE on the audit log table.
  • R-07Audit logs must be transmitted to Sentinel Authority servers in real time or near-real time during the CAT-72 period. Gaps exceeding 15 minutes require justification.
  • R-08System must maintain IP protection: the ENVELO Interlock transmits only operational telemetry. Proprietary system internals are not accessed or transmitted.
  • R-09ODD scope changes after certification initiation require a new scope assessment and may require a new CAT-72 test run.
  • R-10The applicant must designate a primary technical contact responsible for ENVELO Interlock integration and ongoing attestation maintenance.
Certification Process
Phase 1
Application

Submit system information and select the adaptive or prescriptive ODD path. ODD boundaries may be auto-discovered during Phase 3 for the adaptive path.

Phase 2
Scope Assessment

Sentinel Authority reviews the application, defines test parameters, and issues a scope confirmation with associated fee schedule.

Phase 3
Pre-CAT-72 Audit Review

Sentinel Authority conducts an ODDC Audit Control Review — a structured evaluation of ODD boundary definition, ENVELO Interlock enforcement configuration, and telemetry readiness. CAT-72 testing is authorized only upon successful completion.

Phase 4
CAT-72 Testing

72+ hour monitored assessment. Adaptive path begins with learning mode; prescriptive path enforces from the start. All three tiers verified.

Phase 5
Determination

Conformance determination issued with certificate hash and publication to the public registry.

Phase 6
Continuous Surveillance

Certified systems are monitored continuously. Certificates are auto-suspended upon enforcement failure or telemetry loss. Reinstatement requires human authorization. Annual renewal required.
